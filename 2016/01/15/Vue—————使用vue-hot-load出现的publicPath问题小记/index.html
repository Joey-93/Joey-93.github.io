Vue——使用vue-hot-load出现的publicPath问题

简单说明一下问题，webpack-dev-server与webpack Code Splitting共用publicPath造成的路径问题

解决方法已经出来一个了，可以点击这里

工作目录

  • public
    • html
    • build

webpack.config.js中的publicPath,在没有使用vue-hot-load之前，它是被我用来设置Code Splitting之后的js文件的输出路径:

在vue-router中，我使用一下代码来实现代码分割。

调试过程中发现在执行这段require代码时，webpack会根据配置文件中的publicPath用来存放自动分割后的js代码，根据上述配置的publicPath,webpack将会拼接 ../build/ 下的js文件的url到 ./html下 的html中。

目前没问题，能获得编译文件的正确url。

当我配置vue-hot-load的时候，出现了一些我难理解的问题
当我执行webpack-dev-server --inline --hot --no-info这段代码时，结果是

Chrome输出

看得出来，webpack-dev-server是根据publicPath来获取编译后的文件的。然而就是这里，webpack result is served from /../build/
肯定是获取不到，因为正确的路径是

输出结果

Chrome输出

看得出来，这里webpack的hot-load已经出现正常提示了，我尝试了一下，可以实现hot-load。但是Code Splitting之后的路径都是出错的，如果不使用Code Splitting，是可以，但是文件太大，无爱。

我想问问，这种问题该怎么解决，如果不用hot-load的话，编程效率实在太低，需要等待太长时间的编译。



解决方法

额，可能是问题问得太不清楚了，发到Vue论坛都没人鸟~

结果只能我自己继续捣鼓，最后，终于得出了一个解决方案。

回顾上述问题，就两点

  1. 在使用Vue异步组件的过程中(可看上面提供的代码),webpack会根据publicPath的值来拼接script节点到指定的html文件中，实现异步加载vue组件。
  2. 在使用webpack-dev-server的hot-load(热替换)的过程中，webpack会根据publicPath指定的路径来找寻编译后的文件。

看得出来，问题出在publicPath上，根据我上面提供的工作目录可以得出：

工作目录

  • public
    • html
    • build
  • 第一点的publicPath应设置为../build/。
  • 第二点的publicPath应设置为/public/build。

最后我的解决方法是，给webpack-dev-server一个单独的webpack配置文件，也就是把webpack.config.js给它，里面的publicPath是/public/build。

PS:因为我还使用了Grunt布置我的工作流，所以需要额外配置一份webpack配置文件，里面的publicPath值设置为../build/。

这里有一些事忘记说一下

  1. webpack的热替换功能也只是简单的提供代码热替换，也就是说源代码改变，编译代码并不会发生本质改变。所以最终确定结果之后，还需要自己手动再编译一次，才能生成最终的编译文件。这一点我是利用grunt帮我配置任务完成。
  2. webpack的热替换也仅仅适用于像vue组件或者react组件等等需要编译的新开发模式，其他传统的开发模式并不怎么需要。所以不需要折腾的童鞋还是用好基本webpack基础功能再深入也不迟。

总结

webpack是个好东西，这点无误，但是正因为是好东西，所以仅仅是入门是不能体会到它的好处的，这几天也一直看到各处都在说webpack能不能替换gulp、grunt，这点我并不怎么赞同。因为我只把webpack当作一个打包编译工具来使用，而grunt是我用来布置工作流的，两者并不冲突，唯一不足的就是如果要重新部署开发环境到另外一台电脑上，需要npm install 挺多东东的。
